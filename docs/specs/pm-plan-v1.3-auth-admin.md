## 1. アクター定義

| アクター | 略称 | 説明 |
|---------|------|------|
| プラットフォーム管理者 | PF管理者 | PACE運営会社スタッフ。`platform_admin` ロール。顧客個別データ閲覧不可 |
| 組織管理者 | Master | 契約チームの管理者（`master` ロール）。スタッフ・選手管理、チームコード発行 |
| スタッフ | スタッフ | AT / PT / Doctor / S&C。チーム選手のデータ閲覧・承認操作 |
| スタッフ兼選手 | 兼任者 | `staff_members` と `athletes` 両方にレコードを持つユーザー |
| 選手 | 選手 | 自分のデータのみ閲覧・チェックイン入力 |

## 3. ユーザーストーリー + 受入基準

### E1: ログインURL完全分離

#### US-1.1: スタッフ専用ログインページ
**ストーリー:** スタッフとして、`/auth/login` から Magic Link / Google OAuth でログインし、認証後に `/dashboard` へ遷移したい。
**SP: 3** | **優先度: Must**

**受入基準:**
- [ ] `/auth/login` にアクセスすると「スタッフログイン」画面が表示される
- [ ] Magic Link・Google OAuth・メール+パスワードでログイン可能
- [ ] 認証成功後、`/dashboard` にリダイレクトされる
- [ ] 画面下部に「選手の方はこちら」リンクが表示され、クリックで `/auth/athlete-login` へ遷移する
- [ ] 画面は日本語表記（Noto Sans JP）

#### US-1.2: 選手専用ログインページ
**ストーリー:** 選手として、`/auth/athlete-login` から Magic Link でログインし、認証後に `/home` へ遷移したい。
**SP: 3** | **優先度: Must**

**受入基準:**
- [ ] `/auth/athlete-login` にアクセスすると「選手ログイン」画面が表示される
- [ ] Magic Link・Google OAuth でログイン可能
- [ ] 認証成功後、`/home`（GlowingCore）にリダイレクトされる
- [ ] 画面下部に「スタッフの方はこちら」リンクが表示され、クリックで `/auth/login` へ遷移する
- [ ] モバイルファースト設計（タッチターゲット 44x44px 以上）

#### US-1.3: 管理者専用ログインページ
**ストーリー:** PF管理者として、`/auth/admin-login` から認証し、`/platform-admin` へ遷移したい。
**SP: 2** | **優先度: Must**

**受入基準:**
- [ ] `/auth/admin-login` にアクセスすると「管理者ログイン」画面が表示される
- [ ] Magic Link またはメール+パスワードでログイン可能
- [ ] 認証成功後、`/platform-admin` にリダイレクトされる
- [ ] `platform_admin` ロール以外のユーザーがログインした場合、適切なログインページへリダイレクトされる

#### US-1.4: ロール不一致時のミドルウェアリダイレクト
**ストーリー:** システムとして、ログイン済みユーザーのロールとアクセス先が不一致の場合、正しいログインページへリダイレクトし、誤アクセスを防止したい。
**SP: 5** | **優先度: Must**

**受入基準:**
- [ ] `middleware.ts` で認証済みユーザーのロールを検証する
- [ ] 選手ロールが `/dashboard` にアクセス → `/home` へリダイレクト
- [ ] スタッフロールが `/home` にアクセス → `/dashboard` へリダイレクト（ただし兼任者は両方アクセス可）
- [ ] `platform_admin` が `/dashboard` や `/home` にアクセス → `/platform-admin` へリダイレクト
- [ ] 非 `platform_admin` が `/platform-admin/*` にアクセス → 403 または `/auth/login` へリダイレクト
- [ ] 未認証ユーザーは各保護ルートから適切なログインページへリダイレクト

### E2: 選手セルフサインアップ + チームコード

#### US-2.1: チームコード生成・管理
**ストーリー:** Masterとして、選手招待用のチームコードを生成し、有効期限・使用回数上限を設定したい。
**SP: 5** | **優先度: Must**

**受入基準:**
- [ ] 組織管理画面にチームコード管理セクションが追加される
- [ ] コード生成ボタンでランダムな8桁英数字コードが生成される
- [ ] 有効期限を設定可能（デフォルト7日間）
- [ ] 使用回数上限を設定可能（デフォルト無制限）
- [ ] コード一覧に状態（有効/期限切れ/上限到達/手動無効化）が表示される
- [ ] コードの手動無効化が可能
- [ ] `master` ロールのみがコード生成・無効化を実行可能（RLS 適用）

#### US-2.2: 選手セルフサインアップフロー
**ストーリー:** 選手として、`/auth/athlete-login` から新規登録を選択し、Magic Link + チームコードで自分のチームに登録したい。
**SP: 8** | **優先度: Must**

**受入基準:**
- [ ] `/auth/athlete-login` に「新規登録はこちら」リンクが表示される
- [ ] メールアドレス入力 → Magic Link 送信される
- [ ] Magic Link タップ後、チームコード入力画面（`/auth/team-code`）に遷移する
- [ ] 「このコードはチームの管理者から受け取ったものですか？」注意喚起メッセージが表示される
- [ ] 有効なコード入力 → `athletes` テーブルにレコード作成 + `org_id` 紐付け
- [ ] 無効/期限切れ/上限到達コード → エラーメッセージ表示（具体的な理由は開示しない）
- [ ] 登録完了後、`/home` へ遷移する
- [ ] チームコードの使用回数がインクリメントされる

#### US-2.3: チームコードDBスキーマ
**ストーリー:** データエンジニアとして、チームコードの生成・検証・使用追跡に必要なテーブルを設計・マイグレーションしたい。
**SP: 3** | **優先度: Must**

**受入基準:**
- [ ] `team_invite_codes` テーブルが作成される（code, org_id, created_by, expires_at, max_uses, use_count, is_active, created_at）
- [ ] `team_invite_code_uses` テーブルが作成される（code_id, athlete_id, used_at）-- 使用履歴
- [ ] RLS ポリシー: `master` ロールのみ自組織のコード CRUD 可能
- [ ] `code` カラムに UNIQUE インデックス
- [ ] マイグレーションファイルが作成される

### E3: ロール切替スイッチ（方式A）

#### US-3.1: スタッフ兼選手のビュー切替トグル
**ストーリー:** スタッフ兼選手として、スタッフログイン後にヘッダーのトグルで選手ビュー（`/home`）とスタッフビュー（`/dashboard`）を切替えたい。
**SP: 5** | **優先度: Must**

**受入基準:**
- [ ] スタッフログイン後、`athletes` テーブルにも該当ユーザーのレコードがある場合のみ、ヘッダーに切替トグルが表示される
- [ ] トグルは「スタッフビュー / 選手ビュー」のラベル付きスイッチ
- [ ] スタッフビュー ON → `/dashboard` 表示、選手ビュー ON → `/home` 表示
- [ ] 切替時にページ遷移（SPA 内ナビゲーション）
- [ ] 選手ビュー時は自分のデータのみ閲覧可能（RLS: 選手権限として振る舞う）
- [ ] スタッフビュー時は通常のスタッフ権限で動作

#### US-3.2: 選手ログインからのスタッフビュー切替禁止
**ストーリー:** セキュリティ上、選手URLでログインしたユーザーがスタッフビューへ切替えることを禁止し、権限昇格を防止したい。
**SP: 3** | **優先度: Must**

**受入基準:**
- [ ] `/auth/athlete-login` からログインしたユーザーには、たとえ `staff_members` にレコードがあってもトグルが表示されない
- [ ] `/home` 配下のルートのみアクセス可能
- [ ] 直接 `/dashboard` へアクセスした場合、ミドルウェアで `/home` へリダイレクト
- [ ] ログインセッションに `login_source: 'athlete' | 'staff' | 'admin'` を保存する

### E4: platform_admin ロール新設

#### US-4.1: platform_admin ロールのDB・認証基盤
**ストーリー:** システムとして、`platform_admin` ロールをDB・認証・RLSに実装し、プラットフォーム管理画面へのアクセスを制御したい。
**SP: 5** | **優先度: Must**

**受入基準:**
- [ ] `platform_admins` テーブルが作成される（user_id, email, name, created_at, is_active）
- [ ] `is_platform_admin()` RLS ヘルパー関数が作成される
- [ ] `/platform-admin/*` ルートに対するレイアウトレベルの認証ガードが実装される
- [ ] `platform_admin` は `org_id` に紐付かない（組織横断的ロール）
- [ ] `platform_admin` は `athletes`, `staff_members`, `daily_metrics` 等の個別データテーブルに RLS でアクセス不可
- [ ] 集計ビュー（組織別件数・MRR等）のみアクセス可能なビューが作成される
- [ ] マイグレーションファイルが作成される

#### US-4.2: platform_admin 用集計ビュー
**ストーリー:** PF管理者として、顧客個別データにアクセスせず、集計データのみを閲覧したい（情報秘匿性の担保）。
**SP: 5** | **優先度: Must**

**受入基準:**
- [ ] `v_platform_org_summary` ビュー: 組織ID・組織名・契約日・ステータス・スタッフ数・選手数・現在プラン
- [ ] `v_platform_billing_summary` ビュー: 組織ID・組織名・最新請求額・支払い状態・MRR・Dunning状態
- [ ] `v_platform_usage_summary` ビュー: 組織ID・DAU・MAU・チェックイン率・機能別利用件数
- [ ] `v_platform_engine_summary` ビュー: 組織ID・データ蓄積量・推論精度指標・データ品質スコア
- [ ] `v_platform_error_summary` ビュー: エラー種別・件数・直近発生日時
- [ ] 全ビューに `platform_admin` 専用の RLS ポリシーが適用される
- [ ] 個別選手名・個別スタッフ名は含まれない

### E5: プラットフォーム管理画面 P1-P3

#### US-5.1: P1 ダッシュボード
**ストーリー:** PF管理者として、`/platform-admin` で契約チーム数・MRR・未払いアラート件数・システムエラー件数・全体利用率を一覧で把握したい。
**SP: 5** | **優先度: Must**

**受入基準:**
- [ ] `/platform-admin` に5つのKPIカードが表示される: 契約チーム数 / MRR / 未払いアラート件数 / システムエラー件数 / 全体利用率
- [ ] 各カードは前月比の増減を矢印で表示する
- [ ] カードクリックで該当詳細画面（P2-P7）へ遷移する
- [ ] デスクトップレイアウト（最小幅 1024px）
- [ ] データは `v_platform_*` ビューから取得する
- [ ] 自動リフレッシュ間隔: 5分

#### US-5.2: P2 決済状況
**ストーリー:** PF管理者として、組織別のStripe請求一覧・未払い/Dunning状況・MRR推移を確認し、手動請求再送を実行したい。
**SP: 8** | **優先度: Must**

**受入基準:**
- [ ] `/platform-admin/billing` に組織別請求テーブルが表示される（組織名・最新請求額・支払い状態・請求日）
- [ ] 未払い/Dunning状態の組織がハイライト表示される
- [ ] MRR推移グラフ（月次・日次切替可能）が表示される
- [ ] 手動請求再送ボタンが各行に表示される（Stripe API 連携）
- [ ] 請求メモ追加機能がある
- [ ] テーブルはソート・フィルタ可能（ステータス・金額・日付）
- [ ] Stripe ダッシュボードへの外部リンクが表示される

#### US-5.3: P3 契約チーム基礎情報 + プラン管理
**ストーリー:** PF管理者として、契約組織の一覧・ステータス・プラン情報を確認し、プラン変更依頼を管理したい。
**SP: 8** | **優先度: Must**

**受入基準:**
- [ ] `/platform-admin/teams` に契約組織一覧テーブルが表示される（組織名・契約日・ステータス・現プラン・スタッフ数・選手数）
- [ ] ステータスは「アクティブ/休止/解約」で色分け表示
- [ ] 組織行クリック → 詳細パネル展開（プラン変更履歴・チーム規模推移）
- [ ] プラン変更依頼の受付・承認フローUI
- [ ] プラン: Standard / Pro / Pro+CV Addon / Enterprise の4種
- [ ] テーブルはソート・フィルタ・検索可能

### E6: プラットフォーム管理画面 P4-P7

#### US-6.1: P4 システムエラー
**ストーリー:** PF管理者として、APIエラー率推移・エラー一覧・エンジン稼働状況を監視し、障害を早期検知したい。

- [ ] エラー件数が閾値超過時にカードが赤色表示

#### US-6.2: P5 推論エンジン監視
**ストーリー:** PF管理者として、Go/TS切替状況・レイテンシ・Shadow Mode結果を確認し、推論エンジンの品質を監視したい。
**SP: 5** | **優先度: Should**

- [ ] `/platform-admin/engine` に現在のアクティブエンジン表示（Go/TS）

- [ ] レイテンシグラフ（p50/p95/p99、時系列）

#### US-6.3: P6 利用率
**ストーリー:** PF管理者として、組織別DAU/MAU・チェックイン率・機能別利用率を確認し、エンゲージメント状況を把握したい。

- [ ] チェックイン率グラフ（組織別・全体）
- [ ] 機能別利用率（アセスメント・シミュレータ・SOAP等の利用件数）
- [ ] 利用率が低い組織のハイライト（チャーンリスク指標）
- [ ] 期間フィルタ（7d/30d/90d）

#### US-6.4: P7 エンジン成長率
**ストーリー:** PF管理者として、組織別データ蓄積量・推論精度推移・データ品質スコアを確認し、エンジンの成長状況を評価したい。
**SP: 5** | **優先度: Should**

- [ ] 組織別の比較ビュー
- [ ] P3（契約チーム基礎情報）とは独立した画面

## 7. 技術的考慮事項

### DB マイグレーション追加

```sql
-- team_invite_codes テーブル
CREATE TABLE team_invite_codes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code VARCHAR(8) NOT NULL UNIQUE,
  organization_id UUID NOT NULL REFERENCES organizations(id),
  created_by UUID NOT NULL REFERENCES auth.users(id),
  expires_at TIMESTAMPTZ NOT NULL,
  max_uses INT, -- NULL = 無制限
  use_count INT NOT NULL DEFAULT 0,
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- platform_admins テーブル
CREATE TABLE platform_admins (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL UNIQUE REFERENCES auth.users(id),
  email TEXT NOT NULL,
  name TEXT NOT NULL,
  is_active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### ミドルウェア設計方針

```
middleware.ts
├── 未認証チェック → 適切なログインページへ
├── platform_admin チェック → /platform-admin/* 以外をブロック
├── login_source チェック → athlete ログインからの /dashboard アクセスをブロック
├── staff チェック → /dashboard/* へのアクセス許可
└── athlete チェック → /home/* へのアクセス許可
```

### 共通コンポーネント（管理画面用）

- `PlatformAdminLayout` -- サイドバー + ヘッダー（P1-P7 ナビゲーション）
- `KPICard` -- 数値 + 前月比矢印（再利用可能）
- `DataTable` -- ソート・フィルタ・検索対応テーブル
- `TimeSeriesChart` -- Recharts ベースの時系列グラフ
