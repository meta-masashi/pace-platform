# PACE Platform v3.2 実装変更指示書（マスター）

**作成日:** 2026-03-25
**ステータス:** 確定版
**優先度:** 最高（既存のPM計画書・アーキテクチャ仕様を上書き）

---

## 1. プロダクトの核となる変更（方針）

### 1.1 機能の断捨離（デプレリケーション）

以下の機能は、現在のフェーズでは「ノイズ」と判断し、コードベースおよびDBから削除または凍結（非表示）とする。

- **テレヘルス機能**: `/api/telehealth` 関連および `telehealth_sessions` テーブルの廃止。
- **保険請求・請求コード管理**: `/api/billing` 関連および `billing_codes`, `billing_claims` テーブルの廃止。
- **エンタープライズ管理**: `/enterprise` ルートおよび複数チーム一括管理UIの凍結。

### 1.2 強化・維持項目

- **Googleカレンダー同期**: `/api/schedule-events` および OAuth 2.0 連携を維持・強化。スケジュールを「負荷予測のコンテキスト」として利用する。
- **ACWRの維持**: 現場の目安として `daily_metrics` およびUI表示に残す。
- **アラート第一主義**: 異常検知（Critical/Watchlist）をすべてのUIの最優先トリガーとする。

## 2. 【第1部】選手側（Athlete App）の刷新要件

### 2.1 データロジック：ハイブリッド・ピーキング

デバイスの有無を問わず、フィットネス疲労理論に基づく「コンディション・スコア」を算出する。

- **Fitness（フィットネス蓄積）**: 過去42日間の sRPE EWMA。
- **Fatigue（疲労負荷）**: 過去7日間の sRPE EWMA ＋ 朝の主観ペナルティ（睡眠・疲労感不良時）。
- **Readiness（コンディション・スコア）**:

```
(Fitness - Fatigue)
```

を0-100に正規化した数値。

- **Pro Mode（デバイス連携）**: 客観データ（HRV）がベースラインを下回る場合、Fatigueにペナルティ係数を乗算しスコアを下げる。

### 2.2 UI/UX：Ouraインスパイア ＆ プロフェッショナル・クリーン

- **メイン・サークル**: 画面トップに「コンディション・スコア」をリング表示。色は `--teal`（最適）〜 `--red`（回復優先）で変化。
- **インサイトカード**: Gemini API により生成された、専門用語を含まない「日本語の行動アドバイス」を表示。
- **データ・ブレイクダウン**: 以下の3つのカード（`.kpi-card`）を並列配置。
  - **フィットネス蓄積**: 長期トレンド。
  - **疲労負荷**: 直近のストレス。
  - **ACWR**: 0.8〜1.3（安全）〜 1.5（過負荷）を可視化したゲージ付き数値。

## 3. 【第2部】スタッフ側（Staff Web App）の刷新要件

### 3.1 ダッシュボードの再編（YouTube Analytics Style）

`pace_v2.html` のCSSトークンとレイアウトを完全に継承する。

#### 3.1.1 4大KPIカード（`.kpi-row-4`）

1. **Critical アラート**: 即時介入が必要な人数。文字色は `--red`。
2. **プレー可能率 (Availability)**: フル練習参加可能な選手の割合（16/18名 等）。
3. **コンディション・スコア (Team Peaking)**: チーム全体の仕上がり平均値。
4. **Watchlist (隠れリスク)**: 主観と客観のデータ乖離（ギャップ）がある人数。文字色は `--amber`。

#### 3.1.2 分析チャートとスケジュール連携

- **Googleカレンダー同期グラフ**: スケジュール上の「試合（Match）」や「高負荷練習」をX軸にマッピングし、その時点での予想プレー可能率をオーバーレイ表示する。
- **ACWRトレンド**: 過負荷閾値（1.5）に `--amber` の点線を表示し、アラート視認性を高める。

#### 3.1.3 アラート・アクション・ハブ

- **Today's Action**: 異常検知された選手を優先表示。`.rank-dot`（赤・黄）で緊急性を伝える。
- **リスク回避レポート**: AIがメニュー生成時に Hard Lock を適用した場合、`.alert-card.alert-blue` で「重大リスクを未然に防いだ実績」をスタッフに通知する。

## 4. 開発優先順位とマイルストーン

| Phase | 名前 | 内容 |
|-------|------|------|
| **Phase A** | Cleanup | 不要なAPIエンドポイント（Telehealth, Billing）およびDBテーブルの削除・クリーンアップ |
| **Phase B** | Engine | sRPE EWMAベースの「コンディション・スコア」算出ロジックの実装（ハイブリッド対応） |
| **Phase C** | Athlete UI | Oura RingスタイルのサークルUIおよび3大サブ指標カードの実装 |
| **Phase D** | Staff Dashboard | YouTube AnalyticsスタイルのKPI・チャート・アラート表示への刷新 |
| **Phase E** | Calendar Hub | Googleカレンダーのイベントを読み込み、将来の負荷・プレー可能率の予測をダッシュボードに統合 |
