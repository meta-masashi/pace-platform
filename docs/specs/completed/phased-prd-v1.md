## 既存アーキテクチャとの関係

既存 Phase A-E の根幹（推論エンジン `lib/bayes/inference.ts` と DB スキーマ `assessment_nodes`）はそのまま活用。
本 PRD は「コンディション管理＋傷害評価ツール」を **「自律的メニュー生成 OS」** へと引き上げる軌道修正として、Phase A-E の最優先タスクに据える。

## Phase 1：知識の OS 化（EBM-Bayesian Engine 初期実装と UI 独占）

### フェーズ目標

CSV データ群（P0〜F5, A3, A5）をシステムにパースし、「論文エビデンスに基づくリスク推論」と「禁忌/推奨タグによる自律的メニュー生成」を Day 1 から稼働。「7 AM Monopoly（朝の完全独占）」を実現する。

### 1. データスキーマ＆インジェスチョン要件

#### 1-1. Node オブジェクトの定義

CSV の各行を以下の JSON スキーマに変換するインポーターを実装：

| フィールド | 型 | 説明 | 例 |
|-----------|-----|------|-----|
| node_id | string | ノード識別子 | "F3_001" |
| category | string | カテゴリ | "筋力バランス" |
| question_text | string | 質問テキスト（日本語） | "等速性筋力計/HHDで屈曲・伸展筋力測定・比率算出" |
| lr_yes | float | κ補正済み陽性尤度比（`LR_Yes_sr`を使用） | 4.7 |
| lr_yes_clinical | float | 臨床値（表示用、`LR_Yes`） | 4.7 |
| prescription_tags | string[] | 推奨タグ配列 | ["#Str_Hamstring_Eccentric", "#NM_NordicHamstring"] |
| contraindication_tags | string[] | 禁忌タグ配列 | ["!#Sprinting"] |
| evidence_text | string | エビデンス出典 | "Croisier JL et al. AJSM 2008;36:1469" |

**LR 使い分けルール：**

- バックエンド（推論計算）: `LR_Yes_sr`（κ補正値）を使用 — 検者間信頼性を補正した安全な推論
- フロントエンド（表示）: `LR_Yes`（臨床値）を表示 — 論文のオリジナルインパクトを伝達

#### 1-2. ルーティング（依存関係）の構築

`Routing_v4.3` カラムの条件式をパースし、質問分岐と DAG エッジを構築：

- フォーマット例: `Always`, `If P0_002=下半身`, `If P0_001=全身代謝 AND A3_N03=Yes`, `After PS_001`
- 実装: TypeScript の評価関数（正規表現ベースのルールエンジン）でフロントエンドの Wizard UI 表示/非表示を制御

#### 1-3. file_type 拡張

既存: `F1, F2, F3, RTP, MC`
追加: `P0, F4, F5, A3, A5`

### 2. EBM-Bayesian コアエンジン実装要件

#### 2-1. ベイズ更新モジュール（既存 `lib/bayes/inference.ts` を拡張）

```
Post_Odds = Prior_Odds × Node.lr_yes（κ補正値）
Post_Probability = Post_Odds / (1 + Post_Odds)
```

- エンジニアが AI の挙動を完全にトレース可能な決定論的エンジン
- ブラックボックス ML は一切使用禁止

#### 2-2. タグ・コンパイラ（メニュー自律生成アルゴリズム）

リスク確率が閾値（15%上昇）を超えた場合、本日のメニューを自動再構築：

**絶対優先ルール（Conflict Resolution）：**

1. `Contraindication_Tags`（禁忌: `!` マーク）が最強権限
2. `!#ImpactLoad` 発火 → 高衝撃負荷を**強制削除**
3. ブロック枠の代替として `Prescription_Tags` のエクササイズを DB から抽出・挿入

### 3. NLG と UI 要件

#### 3-1. エビデンス・テキスト自動合成（ハイブリッド方式）

**コア生成（決定論的テンプレート）：**

```
[リスク部位]のリスクが[事後確率]に上昇しています。
【根拠】[Node_Name]（[Evidence_Text]）のため。
[Contraindication_Tags]を本日のメニューから除外し、
[Prescription_Tags]を追加しました。
```

**Gemini API 成形（オプション）：**

- テンプレート文字列を Gemini に渡し「医療的根拠・数値・タグを一切変更せずに自然な日本語サマリーに整形」
- フォールバック: Gemini 失敗時はテンプレート文字列をそのまま表示

#### 3-2. 「7 AM Monopoly」ダッシュボード

- 朝一番: 生データ（グラフ・チャート）は**一切表示しない**（スクロール下部のみ）
- 画面100%を「NLG アラート + 修正済みメニュー案」を持つ「本日介入すべき選手カード」で占有

#### 3-3. ワンタップ・アプルーバル

3アクション: **Approve（承認）** / **Edit（修正して承認）** / **Reject（却下）**

**WORM ログ監査（Supabase イミュータブルテーブル）：**

- `approval_audit_logs` テーブル: INSERT のみ許可、UPDATE/DELETE は RLS で完全禁止
- ペイロード: `timestamp, user_id, athlete_id, approved_menu_json, evidence_text_snapshot, data_hash`

### 4. Phase 1 完了の定義（DoD）

1. CSV が全て DB にインポートされノードとして機能
2. モックデータ投入時、正しい LR 計算と禁忌タグブロックが作動
3. エビデンス付き NLG テキスト出力 + Approve で WORM ログ生成
4. ブラックボックス ML が1行もないこと

### ターゲット・リリース

- 対象: プロチーム（タブレット/PC 中心、スタッフ意思決定者）
- スケジュール: 4週間で「P0入力 → ベイズ推論 → 禁忌ブロック → 承認 → WORM保存」の貫通

## Phase 2：データの民主化と「自己進化」機構

### フェーズ目標

スマホ完結型ソリューションで市場を面で制圧。数百万のリアルデータで DAG の確率パラメーターをベイズ更新し「真の Causal AI」を完成。

### 1. エッジ AI モジュール

#### 1-1. Computer Vision パイプライン

- iOS(CoreML) / Android(ML Kit) でマーカーレス姿勢推定
- オンデバイス NPU 処理、JSON（関節角度）のみ送信
- 対象ノード: F1_001(FMS), F2_001(足関節ROM), F4_001(Y-Balance)

#### 1-2. オンデバイス IMU パイプライン

- スマホ腰装着、100Hz 以上で加速度/ジャイロ取得
- イベントドリブン: 3G 超の着地瞬間のみ AI 発火（熱暴走対策）

### 2. ベイズ更新モジュール

#### 2-1. オンライン学習バッチ

- 週次バッチ: 匿名化推論 DB で DAG のノード重み（LR 等）を自動更新
- Ground Truth: 承認ログ + 1-4週間以内の怪我発生/NRS 悪化
- 例: 論文 OR 4.7 → 日本人ユース1万人データで 3.2 に滑らかにベイズ更新

#### 2-2. モデルバージョニング

- v1.0（論文ベース）vs v1.1（データ最適化）のバージョン管理
- フェイルセーフ: 論文値から ±50% 以上逸脱 → 自動停止 + 人間承認フロー

### 3. GTM 連携

- HL7 FHIR 準拠のワンウェイ・エクスポート API（EHR → PACE は対象外）

### Phase 2 DoD

1. iOS/Android で F1_001, F2_001 のマーカーレス角度測定が 60fps で動作
2. 疑似怪我データ 10,000件でベイズ更新が正しく LR 再計算
3. HL7 FHIR 形式エクスポートが正常生成

## Phase 3：時間軸の支配と未来予測（DBN と反事実シミュレーション）

### フェーズ目標

`Time_Decay_λ` と `HalfLife_days` を中枢で稼働させ DBN を完成。Judea Pearl の Do-演算子による反事実エンジンでデジタルツインを提供。

### 1. DBN コアモジュール

#### 1-1. 時間減衰アルゴリズム

```
Risk(t) = Risk(0) × e^(-λt)
```

- `Time_Decay_λ` を用いた日次バッチ計算
- 例: A3_001（HalfLife=60日）→ 数ヶ月の長期負荷管理を自動生成

#### 1-2. 慢性リスクモディファイア

- `Chronic_α_Modifier` 適用: 反復発火選手のリスク減衰を遅延
- 負傷歴の履歴依存性をシステムに実装

### 2. 反事実エンジン

#### 2-1. Do-calculus モジュール

- `P(Y|do(X))` — 因果グラフ上で強制介入時の確率計算
- 単なる条件付き確率 `P(Y|X)` との明確な区別

#### 2-2. What-If ダッシュボード

- 介入スライダー: 練習強度 100% → 60% に変更
- トグルスイッチ: スプリント OFF
- リアルタイム: ハムストリングス肉離れ確率 18% → 4% を即時表示

### 3. 動的自律計画

#### 3-1. 4週間リハビリ・ロードマップの動的リルート

- 毎日の RPE / ROM 推移を監視
- 回復遅延検知 → メニュー強度自動低下 + RTS 後ろ倒し提案
- スタッフはワンタップ承認のみ

#### 3-2. 競技復帰（RTS）予測マッピング

- F5_001(LSI), F5_002(ACL-RSI) の到達確率を時間軸グラフで描画
- 「あと14日で合流可能確率90%」の予測提示

### Phase 3 DoD

1. HalfLife_days に従いリスクスコアが正確に減衰（DBN 証明）
2. トグル操作で Do-演算子が発火しリスク確率がリアルタイム再計算（反事実証明）
3. タスク未完了時に「復帰スケジュール自動再調整」NLG アラートが表示（動的計画証明）

## 技術方針サマリー

| 項目 | Phase 1 | Phase 2 | Phase 3 |
|------|---------|---------|---------|
| 推論方式 | 決定論的ベイズ（DAG） | ベイズ + オンライン学習 | DBN + Do-calculus |
| データ入力 | スタッフ手入力 + CSV | スマホ CV/IMU | 全自動 + リアルタイム |
| NLG | テンプレート + Gemini 成形 | テンプレート + Gemini | Gemini 全文生成 |
| UI | 7AM Monopoly + ワンタップ | モバイル + ワンタップ | What-If ダッシュボード |
| 監査 | WORM ログ（Supabase） | WORM + バージョニング | WORM + 介入ログ |
| 対象 | プロチーム | ユース・アマ・クリニック | メガクラブ・プレミアム |
